# /review — 代码审查（只建议，不改代码）

## 核心约束（必须遵守）

本技能 **只输出审查意见与修改建议**，**不得修改代码或项目文件**：

| 允许                        | 禁止                                             |
| --------------------------- | ------------------------------------------------ |
| `read_file`、`glob`         | `write_file`、`edit_file`                        |
| 只读 `bash`（见下方白名单） | 任何会写入、删除、移动、格式化项目的命令         |
| 在对话中给出「建议怎么改」  | 实际动手改代码、「顺手修一下」                   |
|                             | `spawn_subagent`（可能间接改文件，本技能内不用） |
|                             | 在项目内写 `review.md` 等报告文件                |

**只读 bash 白名单**：

```text
git status
git diff
git diff --staged
git log
git show
git blame
dir /b
type
findstr
where
```

禁止示例：`echo ... >`、`copy`、`move`、`del`、`git add`、`git commit`、`git checkout`、重定向 `>`/`>>`、`pip install`、运行测试/构建（可能产生 `__pycache__` 等产物）。

若用户说「按建议帮我改」，应 **结束本技能**，说明需另开普通任务执行修改；本技能内 **禁止** 进入改代码流程。

## 触发场景

- 用户输入 `/review` 或「审查这段代码」「帮我 code review」
- 指定文件、目录或 `git diff` 范围，**只要建议不要改**
- 合并前评估质量与风险

## 执行步骤

1. **明确范围**：单文件、目录或未提交改动；范围过大时用 `glob` 缩小或请用户指定。
2. **只读收集上下文**：
   - `glob` 定位相关文件
   - `read_file` 阅读目标及直接依赖
   - 必要时 `git diff` / `git diff --staged`
3. **分析**（按 relevance 选重点）：
   - 正确性：逻辑、边界、空值、资源泄漏
   - 安全：注入、路径穿越、敏感信息
   - 可维护性：命名、重复、职责划分
   - 与项目风格是否一致
4. **输出建议报告**（仅文字，不写文件）：

```markdown
## 审查摘要

（1–3 句：总体评价与是否建议合并）

## 修改建议

### [严重] 标题

- 位置：`path:line` 或符号名
- 问题：…
- 建议：…（说明 **应如何改**，不贴完整补丁、不直接改文件）

### [一般] / [建议] …

## 可选改进（非必须）

- …

## 做得好的地方

- …

## 未覆盖范围（如有）

- …
```

每条「建议」必须是 **指导性文字**（例如「将 X 改为 Y」「增加对空列表的判断」），而不是直接提交代码 diff。

5. **收尾**：可问用户是否需要某条建议的详细说明；若需 **落地修改**，提示用户另开任务，本技能到此结束。

## 优先级

| 级别 | 含义                                     |
| ---- | ---------------------------------------- |
| 严重 | 可能导致 bug、安全问题，建议修复后再合并 |
| 一般 | 明确质量问题，建议修复                   |
| 建议 | 风格、可读性、小优化，可选               |

## 禁止事项

- 不要使用 `write_file` / `edit_file` 或任何方式改源码
- 不要创建项目内任何新文件
- 不要替用户实现建议中的修改

## 完成标准

- 用户收到结构化、可操作的 **修改建议**（无代码落地）
- 工作区与审查开始前一致（`git status` 无新增改动）
